# Description

SMTP BDAT (CHUNKING) with a malformed argument: the client sends a numeric
chunk size with an unsupported tail (`BDAT 5 X`). Invalid BDAT syntax must
be treated as recoverable — a decoder event is raised and the command is
kept in the reply queue — instead of a parser error that permanently
disables SMTP parsing for the flow, which would leave every following
message on the connection uninspected.

Redmine ticket: https://redmine.openinfosecfoundation.org/issues/8741

# PCAP

The pcap is generated by the email sandbox scenario
`bdat-bad-tail-next-mail` in `smtp-oddseq.py`.

Postfix rejects the malformed command with a 521 syntax error and closes
the session, so the client pipelines the rest of the dialog in a single
segment which reaches the wire (and the inspector) before the server's
reset lands; none of it is acknowledged.

The important sequence is:

```
<--- 250 CHUNKING

---> mail from:<alice@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<bob@example.com>
<--- 250 2.1.5 Ok
---> BDAT 5 X
<--- 521 5.5.4 Syntax: BDAT count [LAST]

---> BDAT 221 LAST
---> (message body, 221 bytes)
---> mail from:<bob@example.com>
---> rcpt to:<alice@example.com>
---> data
---> (message body)
---> .
```

The test checks that both messages are logged as exactly two transactions
with their SMTP envelope fields, and that the rejected command raises a
decoder event rather than tearing down the parser. Because the client
pipelines past DATA without waiting for its 354 reply, an invalid
pipelined sequence event is also expected.
