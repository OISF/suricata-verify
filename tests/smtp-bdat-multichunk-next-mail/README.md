# Description

SMTP BDAT (CHUNKING) with multiple chunks: the first message is split
across a non-final BDAT chunk and a final BDAT chunk with the LAST
marker, then a second MAIL transaction follows directly with no RSET.
Chunk data is octet counted, so each chunk must be consumed exactly to
its declared boundary and the final chunk and its reply must complete the
first transaction so the second MAIL FROM starts a new transaction.

Redmine ticket: https://redmine.openinfosecfoundation.org/issues/8741

# PCAP

The pcap is generated by the email sandbox scenario
`bdat-multichunk-next-mail` in `smtp-oddseq.py`.

The important sequence is:

```
<--- 250 CHUNKING

---> mail from:<alice@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<bob@example.com>
<--- 250 2.1.5 Ok
---> BDAT 64
---> (message body, first 64 bytes)
<--- 250 2.0.0 Ok: 64 bytes
---> BDAT 154 LAST
---> (message body, remaining 154 bytes)
<--- 250 2.0.0 Ok: 218 bytes queued as <id>

---> mail from:<bob@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<alice@example.com>
<--- 250 2.1.5 Ok
---> data
<--- 354 End data with <CR><LF>.<CR><LF>
---> (message body)
---> .
<--- 250 2.0.0 Ok: queued as <id>
---> QUIT
<--- 221 2.0.0 Bye
```

The test checks that both messages are logged as exactly two transactions
with their SMTP envelope fields, and that the second message's parsed
email headers land in the second transaction.
