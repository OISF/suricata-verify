# Description

SMTP BDAT (CHUNKING) transaction completion: the first message is sent as
a single non-empty BDAT chunk with the LAST marker, then a second MAIL
transaction follows directly with no RSET. The final chunk and its reply
must complete the first transaction so the second MAIL FROM starts a new
transaction instead of merging its envelope into the previous one.

Redmine ticket: https://redmine.openinfosecfoundation.org/issues/8741

# PCAP

The pcap is generated by the email sandbox scenario `bdat-last-next-mail`
in `smtp-oddseq.py`.

The important sequence is:

```
<--- 250 CHUNKING

---> mail from:<alice@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<bob@example.com>
<--- 250 2.1.5 Ok
---> BDAT 221 LAST
---> (message body, 221 bytes)
<--- 250 2.0.0 Ok: 221 bytes queued as <id>

---> mail from:<bob@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<alice@example.com>
<--- 250 2.1.5 Ok
---> data
<--- 354 End data with <CR><LF>.<CR><LF>
---> (message body)
---> .
<--- 250 2.0.0 Ok: queued as <id>
---> QUIT
<--- 221 2.0.0 Bye
```

The test checks that both messages are logged as exactly two transactions
with their SMTP envelope fields, and that the second message's parsed
email headers land in the second transaction.
