# Description

SMTP BDAT (CHUNKING) with a zero-length final chunk: the first message is
just `BDAT 0 LAST`, then a second MAIL transaction follows directly with
no RSET. A zero-length final chunk must complete the first transaction
without entering data mode; it must not consume the following commands as
chunk data or tear down SMTP parsing.

Redmine ticket: https://redmine.openinfosecfoundation.org/issues/8741

# PCAP

The pcap is generated by the email sandbox scenario
`bdat-zero-last-next-mail` in `smtp-oddseq.py`.

The important sequence is:

```
<--- 250 CHUNKING

---> mail from:<alice@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<bob@example.com>
<--- 250 2.1.5 Ok
---> BDAT 0 LAST
<--- 250 2.0.0 Ok: 0 bytes queued as <id>

---> mail from:<bob@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<alice@example.com>
<--- 250 2.1.5 Ok
---> data
<--- 354 End data with <CR><LF>.<CR><LF>
---> (message body)
---> .
<--- 250 2.0.0 Ok: queued as <id>
---> QUIT
<--- 221 2.0.0 Bye
```

The test checks that both messages are logged as exactly two transactions
with their SMTP envelope fields, and that the second message's parsed
email headers land in the second transaction.
