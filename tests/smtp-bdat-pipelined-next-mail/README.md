# Description

SMTP BDAT (CHUNKING) where both chunks of the first message and the
second message's MAIL FROM and RCPT TO are pipelined in a single write,
so they arrive coalesced in one TCP segment. The parser therefore
receives command bytes immediately after a declared chunk payload,
inside the same input buffer, instead of at a segment boundary. Chunk
data is octet counted, so each chunk must be consumed exactly to its
declared boundary and command parsing must resume on the surplus bytes
that follow, completing the first transaction so the coalesced MAIL FROM
starts a second one.

Unlike `smtp-bdat-multichunk-next-mail`, where every BDAT input ends
exactly at its payload boundary, here the boundary falls mid-buffer.

Redmine ticket: https://redmine.openinfosecfoundation.org/issues/8741

# PCAP

The pcap is generated by the email sandbox scenario
`bdat-pipelined-next-mail` in `smtp-oddseq.py`.

The important sequence is:

```
<--- 250 CHUNKING

---> mail from:<alice@example.com>
<--- 250 2.1.0 Ok
---> rcpt to:<bob@example.com>
<--- 250 2.1.5 Ok

# One client write, one TCP segment:
---> BDAT 64
---> (message body, first 64 bytes)
---> BDAT 152 LAST
---> (message body, remaining 152 bytes)
---> mail from:<bob@example.com>
---> rcpt to:<alice@example.com>
<--- 250 2.0.0 Ok: 64 bytes
<--- 250 2.0.0 Ok: 216 bytes queued as <id>
<--- 250 2.1.0 Ok
<--- 250 2.1.5 Ok

---> data
<--- 354 End data with <CR><LF>.<CR><LF>
---> (message body)
---> .
<--- 250 2.0.0 Ok: queued as <id>
---> QUIT
<--- 221 2.0.0 Bye
```

The test checks that both messages are logged as exactly two
transactions with their SMTP envelope fields, that the second message's
coalesced envelope and parsed email headers land in the second
transaction, that no BDAT_CHUNK_LEN_EXCEEDED anomaly is raised, and that
the first transaction is completed by its final chunk's reply
(logged mid-stream) rather than by end-of-flow logging.
